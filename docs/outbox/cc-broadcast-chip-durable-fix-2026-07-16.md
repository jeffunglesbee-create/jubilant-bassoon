## TASK 1 — Investigation

**GOTD auto-detection is real, already wired, and NOT the actual bug.** `build-field-data.js`'s `fetchEspnMlbGotd()` fetches `site.api.espn.com` directly (not `statsapi.mlb.com`) and detects the `'ESPN Unlmtd'`/`'ESPN Unlimited'` signal — exactly the fix TASK 1 hypothesized was needed after the July 2 regression. Confirmed it's called in `main()` (`const espnGotdKeys = await fetchEspnMlbGotd(TODAY)`) and its result threads through `parseMLBFull` → `assignMLBBroadcast` → the output JSON's `espnGOTD` field. The July 2 finding appears to have already been resolved by an intervening session — no new GOTD auto-detection work was needed.

**The precise wrinkle, resolved with real evidence, not assumed:** today's real broadcast is genuinely a *different* ESPN category than GOTD. Live-fetched (via `mcp__FIELD_Handoff__probe_relay_route` against `/mlb-stats/schedule?date=2026-07-16&hydrate=broadcasts(all),team`, the actual relay-proxied `statsapi.mlb.com` route, gamePk `823440`) broadcast entry: `{"name":"ESPN/ESPN App","isNational":true,"callSign":"ESPN",...}` — no `'ESPN Unlmtd'` anywhere. This confirms the codebase's own existing architecture is *correct*, not incomplete: `ESPN_CABLE_SCHEDULE` (national cable exclusive, ~30 games/season, comment: *"'ESPN' in broadcasts() is NOT a reliable signal... appears for many local-affiliate carries too. Authoritative source is the ESPN_CABLE_SCHEDULE lookup"*) and `ESPN_GOTD_SCHEDULE`/`ESPN_GOTD_LOOKUP` (daily streaming pick, `'ESPN Unlmtd'` signal) are genuinely two different real categories, not one signal needing broadening. Today's game (`2026-07-16 NYM@PHI`) is a Cable game — confirmed by `ESPN_CABLE_SCHEDULE`'s own entry, which already had it, correctly dated, before this dispatch. **Broadening the GOTD signal to catch plain 'ESPN' would have been the wrong fix** — it would misclassify a standard national cable broadcast as an ESPN Unlimited streaming exclusive, which it factually isn't.

**`field-data-today.json`'s pipeline runs and reads the right source, but has its own real bug.** Confirmed live: the pipeline ran today (`generated_at: 2026-07-16T09:34:23.332Z`, real file in `outbox/field-data-today.json`) and does read `site.api.espn.com`. But the actual captured output for today's real game shows `"nationalBundle": null, "espnGOTD": false` — despite `ESPN_CABLE_SCHEDULE` having the correct entry. Root cause, traced precisely: `assignMLBBroadcast()`'s `ESPN_CABLE_SCHEDULE` lookup was nested inside `if (bcast.length > 0) { ... if (hasESPN) { ... } }` — gated behind live broadcast-name confirmation from `statsapi.mlb.com`'s `broadcasts(all)` hydration. The workflow runs at 3:30 AM ET, hours before most games' broadcast assignments are attached in that API — so the gate silently blocked the *authoritative* table (which needs no live confirmation; it's a hand-verified ESPN Press Room lookup) from ever being checked. This is the single, precise fix TASK 1 asked me to look for.

## TASK 2 — Fix (durable, not a patch)

Per TASK 2's own branching: GOTD gets real auto-detection (already existed, now actually consumed by the client); Cable keeps its manual table (correctly — real evidence shows no reliable name-based signal exists for it) but gets properly wired everywhere it needs to be, closing two distinct gaps:

1. **`build-field-data.js`, `assignMLBBroadcast()`:** moved the `ESPN_CABLE_SCHEDULE` lookup out of the `hasESPN`/`bcast.length > 0` gate, checked unconditionally (mirroring how the GOTD lookup above it is already unconditional). The table no longer needs live broadcast confirmation to be trusted — restoring its documented "authoritative source" status regardless of build-time data completeness. `hasESPN` (now unused) removed.

2. **`index.html`, `loadMLBSlate()`:** this is the *actual primary, boot-time render path* (`fetchMLBFixtures()` → `loadMLBSlate()` → `normalizeMLBGame()`/`parseBroadcasts()`) — confirmed via tracing `fetchMLBFixtures()`'s own merge logic that it runs independently of `field-data-today.json` and, critically, that the `buildTodaySchedule()` fallback's separate `ESPN_CABLE_SCHEDULE`-aware computation (which already existed and is correct) never survives into the final rendered card: `fetchMLBFixtures()`'s "preserve national TV overrides" merge filters on `g.nationalBundle` truthiness, and that fallback path only ever set `streams`, never `nationalBundle` — so the correctly-computed override was silently dropped and replaced by the ESPN-blind API-sourced version every time. `parseBroadcasts()` deliberately skips `'ESPN'`/`'ESPN+'` broadcasts (correct, to avoid GOTD false positives) but this meant the primary path had *zero* Cable detection of its own.

   Fixed by having `loadMLBSlate()` cross-reference, in priority order: (a) the server's `field-data-today.json` entry for this exact game (`_fieldDataCache.schedules.mlb`, matched by real `home|away` names — the same key convention already used elsewhere in this file for the overlay map) for its auto-detected `espnGOTD`/authoritative `nationalBundle`, then (b) the client's own `ESPN_GOTD_SCHEDULE`/`ESPN_CABLE_SCHEDULE` tables as a fallback when the server data is stale or unavailable. This is durable two ways at once: GOTD games get the server's real auto-detection (no human needs to extend a date table for new GOTD picks going forward), and Cable games get correctly wired into the path that actually renders, with the server-side timing bug fixed so the JSON itself is also correct for any other future JSON consumer.
