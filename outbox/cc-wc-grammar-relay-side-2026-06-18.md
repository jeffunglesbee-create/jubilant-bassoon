BUG 2 — World Cup brief grammar ("1 Goals" / capitalized mid-sentence)
CC-CMD-2026-06-18 — INVESTIGATION ONLY, NO CLIENT FIX

OBSERVED in tonight's WC desk brief (Canada-Qatar context):
  - "after each scored 1 Goals this season"
  - "following Qatar's 0 Goals this season"

Issues:
  - "Goals" capitalized mid-sentence (should be lowercase)
  - "1 Goals" — wrong pluralization (should be "1 goal")
  - "0 Goals" — should be "no goals" or "0 goals" (lowercase)

INVESTIGATION (Rule 1 / Rule 48):

Audited every templated path in index.html that could produce a string
containing capital-G "Goals" alongside a numeric prefix:

  $ grep -nE "Goals" index.html
  $ grep -nE "scored.+goals|goals scored" index.html
  $ grep -nE "\\\$\\{[^}]*goal" index.html
  $ grep -nE "GF[^A-Z]|GF$" index.html (column label)

Results:
  - One only-comment hit ("Goals come in bunches.") — not a template.
  - Two PL final-day fallback strings ("goals scored" — lowercase, fine).
  - GF appears as a standings-table column label and inside FD/MLS
    standings context blocks — never rendered to prompt as "Goals".
  - matchupNote strings for Canada and Qatar do not contain "Goals" with
    capital G.
  - buildGameStandingsContext / buildH2HSummary / buildScoreNarrativeContext
    all emit lowercase or no goal-stat text.

Conclusion: the offending string is NOT in index.html.

ACTUAL OWNER — field-relay-nba (separate repo):

The WC tournament/match briefs are generated by the relay:
  - GET /wc/brief/tournament  → tournament brief
  - GET /wc/brief/<gameId>    → per-match brief
  - Async fetched and injected client-side in _fetchWCTournBriefForSchedule
    (index.html:11926) and the per-game injection at index.html:16447.

The relay's prompt-context builder owns the "1 Goals this season" string.
Likely sites in the relay:
  - A prompt template that injects team form / GF totals with a "Goals:"
    or "${n} Goals" label
  - A WC standings → prose helper that emits "X Goals" instead of "X goal/goals"

RECOMMENDED RELAY-SIDE FIX (handoff to a relay session):

In the WC prompt context builder, replace any pattern like:
    `${gf} Goals`
with:
    `${gf} goal${gf === 1 ? '' : 's'}`

And in any prompt instruction text that names the stat, write "goals"
(lowercase, mid-sentence) so the LLM follows convention.

CLIENT-SIDE MITIGATION CONSIDERED — none applied:

Considered a client-side post-processor that lowercases " Goals" → " goals"
and rewrites " 1 Goals" → " 1 goal" on the brief text after it lands. Did
not implement: (a) violates Rule 60 (relay owns the contract — fix at the
source, not via client compensation), (b) it would mask the bug from the
relay session that needs to fix it, (c) it would also lowercase legitimate
capitalised "Goals" if any (e.g. "Goals Saved Above Expected" → "goals
saved above expected", which is wrong).
