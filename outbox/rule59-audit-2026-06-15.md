### 1a. Layer 1 — null-score guard in `fetchV2AllScores`

**Audit description:** "when `_scoresNull && !prev`, the espnScores write is skipped."

**Reality (index.html:15367-15378):**
```js
espnScores[key] = {
  ...v2Entry,
  homeScore: (v2Entry._scoresNull && prev?.homeScore) ? prev.homeScore : v2Entry.homeScore,
  awayScore: (v2Entry._scoresNull && prev?.awayScore) ? prev.awayScore : v2Entry.awayScore,
  …
};
```

This is a **merge guard, not a skip.** The write always proceeds. The ternary only preserves the prior score WHEN `prev` already has one. If `_scoresNull && !prev`:
- `prev?.homeScore` is `undefined` → ternary takes the right branch → `v2Entry.homeScore` is used.
- `v2Entry.homeScore` is `0` (mapV2ToESPN at line 15212 normalises null to 0 via `?? 0`).
- **Result: a `0-0` entry IS written when `_scoresNull && !prev`.**

The audit's "write is skipped" description is inaccurate. This is a Rule 48 Class D root-cause assumption error in the audit text.

**Other 0-0 writers on the same key?** I scanned for additional writers:
- `_scoresBySource[key].apisports` at line 15384-15400 mirrors the same merge logic → can also store `0-0` when no prev (line 15390-15391).
- ESPN-native writer at `fetchESPNScores` (line 17567) writes its own `espnScores[key]` with `homeScore`/`awayScore` directly from the ESPN response. Not gated by `_scoresNull` because the ESPN path doesn't produce that field. If ESPN returns 0-0 for a finished game, this path stamps 0-0 in.
- `hydrateEspnScoresFromFinals` at line 33293 writes only when there is no existing entry OR the existing entry is blank — it does not corrupt non-blank entries.
- `saveEspnFinal` at line 33112 writes to localStorage (not `espnScores` directly) so it's an upstream source for Layer 2 but not a direct overwrite vector.

**Finding:** at least two paths can plant a `0-0` post entry — the V2 merge path when `!prev`, and the ESPN-native writer if the ESPN response is wrong. Layer 1 reduces but does not eliminate the failure mode.
